Introduction

In the dust and sand surrounding the vault, you unearth a rusty PCB. You try to read the etched print — it says Open..W...RT, a router! You hand it over to the hardware gurus and to their surprise the ROM chip is intact. They manage to read the data off the tarnished silicon and give you back a firmware image. It's now your job to examine the firmware and recover useful information that will help unlock and bypass the vault's countermeasures.

This challenge gives us a raw binary dump of a router's flash chip. Our goal is to extract version numbers, credentials, network configs, and port forwarding rules. Think of it like finding a hard drive on the street — we're going to explore every layer of what's on it.

// KEY CONCEPT — What is Firmware?

Firmware is the permanent software stored on a hardware chip. For a router, it contains the entire operating system, configuration files, and all user settings. When you dump a chip, you get one big binary blob containing all of this compressed and packed together.

Step-by-Step Walkthrough
01
Identify the File
Reconnaissance — what format are we dealing with?

Always start by asking: what kind of file is this? The file command reads the first few bytes (the "magic header") and identifies the format — without relying on the filename extension at all.

BASH
file chal_router_dump.bin

# Output:
chal_router_dump.bin: data

It came back as generic data — meaning it's a raw binary blob with no recognisable standard header. This is completely normal for flash chip dumps. The file is 16 MB (0x1000000 bytes), exactly matching the flash size of typical budget home routers.

// TOOL: file

The file command identifies file types by reading magic bytes at the start of the file, not by relying on the filename extension. It's the fastest first-look tool in your forensics toolkit — always run it first.

02
Extract Readable Strings
Mining plain-text data from inside the binary

The strings command pulls out any sequence of printable characters from a binary file. Even compiled, compressed code has version strings, copyright notices, error messages, and config values baked directly into it — completely in the clear.

BASH
strings chal_router_dump.bin | grep -i "openwrt"

# Key output lines:
MIPS OpenWrt Linux-5.15.134
OpenWrt kernel loader for MIPS based SoC

This immediately reveals our first answer: kernel version 5.15.134. We also learn it's a MIPS architecture device — common in budget routers using MediaTek SoCs (TP-Link, ASUS, Xiaomi budget lines).

// WHY DOES THIS WORK?

Compilers embed version strings, copyright notices, and debug messages as plain text into binaries even after compilation. The strings tool finds any run of 4+ printable ASCII characters — no decryption, no disassembly needed. One of the highest-signal, lowest-effort recon tools available.

03
Find the Filesystem — Magic Bytes
Locating and carving the SquashFS partition

Router firmware contains a compressed filesystem packed somewhere inside the binary blob. The most common format for OpenWRT is SquashFS — a read-only compressed filesystem designed specifically for embedded devices. Every binary format starts with a unique byte sequence called magic bytes. SquashFS uses hsqs (little-endian) or sqsh (big-endian).

We write a quick Python script to scan the binary and find it:

PYTHON
# Open the firmware as raw bytes
data = open('chal_router_dump.bin', 'rb').read()

# Scan for SquashFS magic bytes (little-endian)
pos = data.find(b'hsqs')
print(f"Found SquashFS at offset {pos} ({hex(pos)})")

# Output:
Found hsqs at offset 4375240 (0x42c2c8)

The filesystem starts at byte 4,375,240 into the file. Now we carve it out using dd — a precision binary cutter — then extract the contents with unsquashfs:

BASH
# Carve from offset to end of file (bs=1 = byte-by-byte precision)
dd if=chal_router_dump.bin of=squashfs.img bs=1 skip=4375240

# Extract the entire SquashFS into a folder called 'extracted/'
unsquashfs -d extracted squashfs.img

# Output:
created 1010 files, 114 directories, 203 symlinks
// NOTE: dd bs=1 is slow on large files

Using bs=1 skip=N copies byte-by-byte, which is accurate but slow for 16 MB files. For speed, use a larger block size with calculated skip/count values — or use binwalk -e which handles identification and extraction automatically.

04
Explore the Filesystem
Navigating the extracted Linux directory tree

The extracted folder is a complete Linux filesystem — identical to what's running on the physical router right now. We browse it exactly like any Linux system. Key locations to know:

extracted/ ├── bin/ — system binaries (ls, cat, sh...) ├── etc/ ★ config files live here │ ├── openwrt_release → OpenWRT version string │ ├── shadow → password hashes (factory default) │ ├── passwd → user accounts │ └── config/ │ ├── network → IP addresses, PPPoE credentials │ ├── wireless → WiFi SSID & password │ └── firewall → port forwarding rules ├── lib/modules/ │ └── 5.15.134/ → kernel version confirmed in folder name └── www/ — LuCI web interface files
BASH
cat extracted/etc/openwrt_release

DISTRIB_ID='OpenWrt'
DISTRIB_RELEASE='23.05.0'
DISTRIB_REVISION='r23497-6637af95aa'
DISTRIB_TARGET='ramips/mt7621'
DISTRIB_ARCH='mipsel_24kc'

Answer confirmed: OpenWRT version 23.05.0. The target ramips/mt7621 tells us it's running on a Ralink/MediaTek MT7621 chipset — one of the most common budget router SoCs in the world.

Checking etc/shadow in the base SquashFS layer shows root with no password — but that's just the factory default image. The real, user-set credentials live in the writable overlay filesystem, covered in the next step.

05
The Hidden Layer — JFFS2 Overlay
Where the real user data is stored

This is the most critical step in the entire challenge. OpenWRT uses a two-layer filesystem architecture. Miss this layer, and you miss everything that matters.

Layer Format Purpose Contains
Base SquashFS Read-only factory image Default configs, system binaries
Overlay JFFS2 Writable user changes Changed passwords, custom configs, keys
// COMMON MISTAKE

If you only analyse the SquashFS base layer, you'll see factory defaults — blank passwords, empty configs. The actual user-configured data lives entirely in the JFFS2 overlay. This is the single most common oversight when analysts do firmware triage for the first time.

We scan for the JFFS2 partition, which begins after the SquashFS data ends. JFFS2 nodes use the magic bytes 0x1985 (stored little-endian as \x85\x19):

PYTHON
# JFFS2 little-endian node magic: 0x1985
jffs2_magic = b'\x85\x19'

# Search only past where SquashFS ends
pos = data.find(jffs2_magic, after_squashfs)
print(f"JFFS2 found at {hex(pos)}")

# Output:
JFFS2 node at 0x7c0000
BASH
# Carve the JFFS2 partition starting at offset 0x7c0000
dd if=chal_router_dump.bin of=overlay.jffs2 bs=1 skip=$((16#7c0000))

# Extract using jefferson  (pip install jefferson)
jefferson overlay.jffs2 -d jffs2_out

# The overlay stores all user changes inside a sysupgrade tarball
tar -xzf jffs2_out/upper/sysupgrade.tgz -C /tmp/overlay/

Now /tmp/overlay/etc/ contains the real, live configuration files with all actual credentials. Time to read them.

06
Extracting the Credentials
Reading secrets from the JFFS2 overlay configs

Root password hash — from /tmp/overlay/etc/shadow:

BASH
cat /tmp/overlay/etc/shadow

root:$1$YfuRJudo$cXCiIJXn9fWLIt8WY2Okp1:19804:0:99999:7:::

The $1$ prefix means MD5crypt — a weak hashing algorithm by modern standards. This hash can be cracked offline using hashcat or john the ripper with a common wordlist.

PPPoE WAN credentials — from /tmp/overlay/etc/config/network:

UCI CONFIG
config interface 'wan'
    option device   'wan'
    option proto    'pppoe'
    option username 'yohZ5ah'
    option password 'ae-h+i$i^Ngohroorie!bieng6kee7oh'
    option ipv6     'auto'

WiFi configuration — from /tmp/overlay/etc/config/wireless:

UCI CONFIG
config wifi-iface 'default_radio0'
    option ssid       'VLT-AP01'
    option encryption 'sae-mixed'   # WPA3/WPA2
    option key        'french-halves-vehicular-favorable'

Port forwarding rules — from /tmp/overlay/etc/config/firewall:

UCI CONFIG
config redirect          # Database server
    option src_dport '1778'  →  192.168.1.184:5881

config redirect          # Web server
    option src_dport '2289'  →  192.168.1.119:9889

config redirect          # NAS / storage
    option src_dport '8088'  →  192.168.1.166:4431
// REAL-WORLD IMPLICATION

PPPoE passwords, WiFi keys, and other secrets are stored as unencrypted plaintext in UCI config files. This is a genuine real-world risk — physical access to a router means full credential recovery. No soldering required if the bootloader is accessible via the serial console.

Tools Used
🔧
Tools Used
strings

Extracts printable text sequences from any binary. Built into Linux/macOS. Fastest first-recon tool for firmware analysis.

Python 3

Used to scan for magic bytes at arbitrary binary offsets — far more flexible than grep for raw binary data.

dd

Unix block-level copy utility. Used to carve the SquashFS and JFFS2 partitions from precise byte offsets in the firmware blob.

unsquashfs

Part of the squashfs-tools package. Decompresses and extracts SquashFS filesystem images into a browsable directory tree.

jefferson

Python-based JFFS2 filesystem extractor. Install via pip install jefferson. Essential for reading OpenWRT overlay partitions.

tar

Standard archive tool. Used to unpack the sysupgrade.tgz tarball that wraps the JFFS2 overlay's user config files.

Key Takeaways
💡
Key Takeaways
01 — Firmware is just a file

A chip dump is a regular binary file. You can search it, slice it, and extract filesystems from it using standard Linux tools — no special hardware needed once you have the dump. Everything is accessible at the byte level.

02 — Magic bytes reveal structure

Every embedded format (SquashFS, JFFS2, gzip, U-Boot, etc.) starts with a unique byte signature. Scanning for these tells you exactly where each partition begins — no partition table needed.

03 — Always check the overlay

OpenWRT and many embedded Linux systems use a layered filesystem (OverlayFS). The base SquashFS layer has factory defaults; the JFFS2 overlay has real user data. Missing the overlay means missing passwords, custom configs, and keys.

04 — Configs store credentials in plaintext

PPPoE passwords, WiFi keys, and other secrets are stored as unencrypted plaintext in UCI config files. This is a genuine real-world security risk. Physical access to a router — or a firmware dump — means full credential recovery with zero cracking required.
